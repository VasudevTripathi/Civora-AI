# ADR-002: Backend API Framework — FastAPI

## Decision
We will use **FastAPI** as the web framework for the Python API services.

## Context
Civora AI needs to expose fast, typed REST endpoints for the frontend application. It also requires automated documentation (OpenAPI/Swagger) to maintain compliance and facilitate developer integration. High concurrent API capability is necessary for streaming RAG responses from Gemini and handling multiple simultaneous file uploads.

## Alternatives Considered
* **Flask**: Simple and highly flexible, but lacks native type validation and asynchronous support. It requires multiple external libraries (Flask-RESTful, Marshmallow) to achieve what FastAPI handles natively.
* **Django / Django REST Framework**: Features a powerful ORM and admin dashboard out of the box, but is highly monolithic, slow to compile, and has a synchronous-first design which conflicts with asynchronous AI streaming routines.

## Consequences
* **Positive**:
  * Out-of-the-box support for async request processing, optimizing I/O bound database and AI requests.
  * Automated Pydantic-based data serialization, validation, and interactive OpenAPI documentation.
  * High execution speed matching Go and Node.js performance.
* **Negative**:
  * Requires explicit type hinting across all routers, and requires careful handling of thread pools when calling blocking synchronous libraries.
