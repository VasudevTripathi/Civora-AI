# ADR-003: Relational Database — PostgreSQL

## Decision
We will use **PostgreSQL** as the primary relational database for transactional business data.

## Context
Civora AI must manage ACID-compliant records for user accounts, billing transactions, corporate structures, and filing statuses. Because our data models have clear relational entities (e.g. one User has many Businesses, which have multiple Applications), we require a robust relational database engine with strong integrity checks, transactional security, and scale capabilities.

## Alternatives Considered
* **MySQL**: Competent relational engine, but lacks PostgreSQL's advanced SQL compliance, custom index types (like GIN/GiST), and native JSONB capability (which is useful for storing unstructured government metadata).
* **MongoDB**: A document store that is highly elastic, but lacks foreign key integrity constraints, transactional controls across collections, and standardized query tools, which increases the risk of data corruption for complex legal filings.

## Consequences
* **Positive**:
  * Industry-standard ACID transactional properties ensuring complete reliability for filing data.
  * Exceptional support for JSONB, enabling us to store semi-structured compliance forms.
  * Extensible via plugins (e.g. pgvector) if we want to colocate simple semantic search in the primary DB later.
* **Negative**:
  * Requires database connection pooling configurations to prevent exhaustion of connection slots under high concurrent async API requests.
