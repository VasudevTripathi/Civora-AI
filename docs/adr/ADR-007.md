# ADR-007: Data Access Design — Repository Pattern

## Decision
We will isolate our database operations using the **Repository Pattern**.

## Context
In Clean Architecture, core business services must not run database query code directly. Direct SQL or ORM calls (e.g. `session.exec()`) link the business logic to the database engine. If queries are mixed directly inside the services, we cannot test services without mocks of the database session, and swapping DB systems or optimizing query structures becomes difficult.

## Alternatives Considered
* **Active Record Pattern (Direct ORM usage)**: Simpler to write, but creates a hard dependency between domain logic and the database library.
* **Service-Direct SQL**: Highly anti-pattern in enterprise systems, leading to scattered SQL queries, code duplication, and SQL injection risks.

## Consequences
* **Positive**:
  * Decouples domain logic from database query syntax.
  * Simplifies testing: we can inject a mock repository class that returns dummy list entities in memory without using a running test database.
  * Centralizes database transaction and optimization code.
* **Negative**:
  * Adds an additional abstraction layer (each model requires an interface repository class and a concrete SQL implementation class), which increases folder complexity.
