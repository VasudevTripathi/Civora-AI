# ADR-001: Architectural Pattern — Clean Architecture

## Decision
We will design the Civora AI backend services following the principles of **Clean (Onion) Architecture**.

## Context
Government regulations, compliance requirements, and filing form layouts change frequently. Meanwhile, core database choices, framework updates, or external AI models might evolve or be swapped out. If our core business rules (e.g. calculation of franchise taxes, evaluation of registration state conditions) are tightly coupled with external HTTP libraries (like FastAPI), database clients (like SQLModel), or LLM SDKs, changing state laws or database layers would cause massive ripple effects, leading to instabilities.

## Alternatives Considered
* **Transactional Script / MVC**: Simplest to implement initially. However, it quickly becomes unmaintainable because business rules are scattered across controllers (API routes) and ORM models, breaking the separation of concerns.
* **Microservices**: Isolates boundaries but introduces heavy operations, network latency, distributed transactions, and configuration overhead, which is premature for our current stage.

## Consequences
* **Positive**:
  * The business domain logic has zero external dependencies and can be tested purely in memory without database connections or web servers.
  * Swapping infrastructures (e.g., changing SQL databases or vector databases) requires implementing a new adapter interface without touching core business rules.
* **Negative**:
  * Requires more boilerplate files initially (schemas, interfaces, models, controllers) which increases cognitive overhead for new developers.
