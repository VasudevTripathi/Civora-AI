# ADR-004: Knowledge Architecture — Knowledge Graph

## Decision
We will model our government compliance metadata using a **Knowledge Graph** paradigm.

## Context
Government regulations do not operate in isolation. A business activity (e.g. serving food) is governed by specific laws, which define licenses, which require specific documents, which are issued by distinct municipal departments, which operate in specific states or districts. A standard relational model requires complex, multi-join queries to represent these regulatory trees. Using a Knowledge Graph allows us to explicitly define and navigate relationships (e.g., `License -> issued_by -> Authority -> located_in -> State`).

## Alternatives Considered
* **Flat Relational Schemas**: High complexity to modify or extend. Adding new categories of regulatory requirements requires modifying schemas and writing complex migration SQL scripts.
* **Hierarchical Document Trees**: Simple to read, but fails when relationships are cross-cutting (e.g. a document like "Proof of Address" is required by multiple independent licenses across different departments).

## Consequences
* **Positive**:
  * Highly extensible. Adding a new state, law, or business activity only requires adding new nodes and edges to the graph configuration, not writing code.
  * Allows graph-traversal compliance path discovery: we can query the shortest path of licenses and dependencies a user needs.
  * Provides perfect grounding context for RAG queries: we can extract sub-graphs to guide Gemini's context assembly.
* **Negative**:
  * Requires specialized graph indexing patterns. To keep our stack simple, we will represent the graph nodes and edges in our PostgreSQL database using standardized relational mapping models rather than introducing a separate complex graph DB (like Neo4j) too early.
